> 本文档用于面试讲解，包含设计决策的"为什么"

## 一、项目背景与价值

### 1.1 业务场景

```
用户上传视频 → 系统异步处理 → AI理解内容 → 生成结构化总结
```

**典型应用场景**：
- 视频会议纪要自动生成
- 在线教育课程摘要
- 监控视频智能分析
- 短视频内容审核

### 1.2 技术挑战（面试重点）

| 挑战 | 具体问题 | 解决思路 |
|------|---------|---------|
| **大文件上传** | 视频文件GB级别，网络中断需重传 | 分片上传 + 断点续传 |
| **处理耗时长** | AI分析可能耗时数分钟 | 异步处理 + 消息队列解耦 |
| **高并发** | 多用户同时上传和处理 | 限流 + 背压控制 |
| **可靠性** | 处理失败需重试，不能丢任务 | 持久化队列 + 重试机制 |
| **成本控制** | AI调用昂贵 | 预估成本 + 配额管理 |

### 2.2 为什么这样分层？（面试话术）

> **面试官可能问**：为什么要拆成 API 和 Worker 两个服务？

**回答要点**：

1. **职责分离**：
   - API服务：面向用户，要求低延迟，无状态，可快速水平扩展
   - Worker服务：面向计算，CPU/GPU密集，可独立扩缩容

2. **故障隔离**：
   - AI分析可能因为外部服务慢而阻塞
   - 如果合并部署，会拖慢用户上传请求的响应

3. **资源利用率**：
   - API服务可以部署在普通机器
   - Worker服务可以部署在GPU机器

4. **独立扩展**：
   - 任务积压时，只扩Worker节点
   - 上传高峰时，只扩API节点

### 4.2 为什么用Kafka而不是RabbitMQ？（面试必问）

| 维度 | Kafka | RabbitMQ | 本项目选择 |
|------|-------|----------|-----------|
| **吞吐量** | 百万级/秒 | 万级/秒 | ✅ Kafka |
| **消息持久化** | 顺序写磁盘，高效 | 内存为主 | ✅ Kafka |
| **消息回溯** | 支持重复消费 | 消费后删除 | ✅ Kafka |
| **延迟** | 毫秒级 | 微秒级 | 可接受 |
| **适用场景** | 日志、大数据、流处理 | 传统业务消息 | ✅ 本项目 |

> **选择Kafka的理由**：
> 1. 视频分析任务量大，需要高吞吐
> 2. 任务处理失败需要重试，Kafka支持消息重放
> 3. 未来可能接入Flink做实时分析，Kafka天然集成

### 5.2 为什么这样设计表结构？（面试话术）

> **问题1：为什么upload_session表的uploaded_chunks用JSON而不是关联表？

**回答**：
1. 分片数量有限（通常几十到几百个），JSON存储完全够用
2. 避免多表关联查询，提升性能
3. 单次查询获取所有信息，减少数据库交互
4. 使用MySQL 5.7+的JSON函数可以方便查询

> **问题2：analysis_task表为什么要有retry_count字段？

**回答**：
1. AI服务调用可能因网络抖动失败，需要重试
2. 记录重试次数，超过阈值需要人工介入或降级处理
3. 用于监控告警，频繁重试说明系统有问题

## 六、关键技术决策

### 6.1 分片大小选择

| 分片大小 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 1MB | 速度快，失败代价小 | 请求数多，服务端压力大 | 网络不稳定 |
| **5MB** | 平衡点 | - | ✅ 推荐选择 |
| 10MB | 请求数少 | 失败重传代价大 | 网络稳定 |

**为什么选择5MB**：
1. AWS S3、阿里云OSS推荐的分片大小
2. 1GB视频 = 200个分片，并发上传效率高
3. 单个分片上传失败，重传代价可接受

### 6.2 Kafka分区策略

```java
// 按用户ID分区，保证同一用户的任务顺序
public class UserIdPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        // key = userId
        int partitionCount = cluster.partitionCountForTopic(topic);
        return Math.abs(key.hashCode()) % partitionCount;
    }
}
```

**为什么按userId分区**：
1. 保证同一用户的任务按提交顺序处理
2. 便于问题排查（某用户的所有任务在同一分区）
3. 实现用户级别的消费限流

### 6.3 限流策略

| 层级 | 限流位置 | 限流算法 | 阈值 |
|------|---------|---------|------|
| **网关层** | API Gateway | 滑动窗口 | 100 QPS/用户 |
| **应用层** | 上传接口 | 令牌桶 | 10 上传/分钟/用户 |
| **消费层** | Kafka Consumer | 信号量 | 5 并发/Worker |

**为什么用多种限流算法**：
1. 滑动窗口：精确控制QPS，防止瞬时流量
2. 令牌桶：允许一定程度的突发流量，用户体验更好
3. 信号量：控制并发数，防止资源耗尽

### 9.2 可能被问到的问题

1. **Q: 如果Kafka挂了怎么办？**
   - A: Kafka是多副本集群，单个Broker宕机不影响服务
   - 消息持久化到磁盘，重启后自动恢复

2. **Q: 如何保证消息不丢失？**
   - A: 生产者开启acks=all，消费者手动提交offset

3. **Q: 如何避免重复消费？**
   - A: 消费者幂等设计，基于taskId去重

4. **Q: 大促期间如何扩展？**
   - A: API服务无状态，水平扩展；增加Kafka分区数；增加Worker节点
